# Filter gpt-5.4-mini from model picker

## What changed

Added an explicit `UNAVAILABLE_MODELS` denylist in `lib/pi-model-registry.server.ts` and
applied it as a new R0 pre-filter in `filterToLatestVersions`.

The initial entry in the denylist is `gpt-5.4-mini`, which the pi SDK model registry
includes but OpenAI's API rejects with:

```
400 model "gpt-5.4-mini" is not supported
```

## Why

The pi SDK's `models.generated.js` tracks forthcoming models (gpt-5.4-mini, etc.) before
they are actually deployed by the provider.  The Primordia model picker reads the SDK
registry at runtime and uses R1–R4 filters to produce a clean list.  R3 intentionally
keeps "mini" variants (gpt-4.1-mini, o4-mini are valid), so gpt-5.4-mini was kept and
surfaced to users.  When they selected it and ran an evolve session, the pi worker hit
the 400 and the session errored immediately.

The denylist approach is the smallest targeted fix: we can't automatically detect
"not yet released" from registry metadata, but we can block by known-bad ID and remove
the entry once the model becomes available.

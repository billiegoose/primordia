# Add exe.dev account login option

## What changed

Added a third login/signup method to the `/login` page that lets users authenticate using their exe.dev account.

### New files

- **`app/api/auth/exe-dev/route.ts`** — A GET route that handles the exe.dev login flow:
  1. If the `X-ExeDev-Email` header (injected by the exe.dev HTTP proxy) is present, it finds or creates a Primordia user whose username is that email address, issues a session cookie, and redirects to the intended destination.
  2. If the header is absent (user not yet authenticated with exe.dev), it redirects to `/__exe.dev/login` so the proxy can authenticate them, then sends them back to the same route to complete sign-in.

### Modified files

- **`app/login/page.tsx`** — Now reads the `X-ExeDev-Email` header server-side and passes it as an `exeDevEmail` prop to `LoginClient`, enabling the tab to show the already-known email when the proxy has authenticated the user.

- **`app/login/LoginClient.tsx`** — Added a third tab labelled **"exe.dev"** alongside the existing "Passkey" and "QR Code" tabs:
  - If `exeDevEmail` is set (proxy headers present), the tab shows the email and a one-click "Sign in as {email}" button.
  - If not set, the tab shows a "Sign in with exe.dev" button that navigates to `/api/auth/exe-dev`, which redirects through the exe.dev login page before returning.
  - Added `ExeDevIcon` (a chevron SVG) for the button.
  - Updated the page subtitle to mention exe.dev.

## Why

Users who run Primordia on exe.dev servers already have an exe.dev account. This gives them a frictionless third login path — no passkey setup required — and uses their email address as their Primordia username, keeping identity consistent with their exe.dev account.

The implementation follows the existing header-injection mechanism documented by exe.dev rather than OAuth, keeping the code minimal (no new dependencies, no client secrets to manage).

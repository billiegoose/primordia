# Prefer exe.dev LLM gateway with API key fallback

## What changed

- Added `lib/llm-client.ts` — a shared factory that creates an Anthropic SDK client configured to use the **exe.dev LLM gateway** (`http://169.254.169.254/gateway/llm/anthropic`) when running on an exe.dev VM, or falls back to `ANTHROPIC_API_KEY` when the gateway is not reachable.
- The gateway is probed once per server process with a 2-second timeout (link-local address is unreachable outside exe.dev, so the fallback activates quickly). The result is cached for the lifetime of the process.
- The active backend is shown at the top of every evolve session's progress log (e.g. `> LLM backend: exe.dev gateway`) so it is visible in the UI rather than buried in server console output.
- `app/api/chat/route.ts` and `app/api/evolve/route.ts` now call `getLlmClient()` instead of constructing `new Anthropic()` directly.
- `app/api/check-keys/route.ts` now reports `ANTHROPIC_API_KEY` as missing **only** when the gateway is also unavailable — so the "missing keys" warning is suppressed on exe.dev where no API key is needed.
- `PRIMORDIA.md` updated to reflect the conditional status of `ANTHROPIC_API_KEY` and the gateway-first approach.

## Why

The exe.dev platform includes a built-in LLM gateway that covers API costs from each VM's monthly token allocation, eliminating the need to provision and rotate an Anthropic API key for deployments on exe.dev. By preferring the gateway and transparently falling back to the API key, the app works without configuration on exe.dev and continues to work everywhere else.

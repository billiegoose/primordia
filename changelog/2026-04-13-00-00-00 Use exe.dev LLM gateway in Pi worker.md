# Use exe.dev LLM gateway in Pi worker

## What changed

`scripts/pi-worker.ts` now probes the exe.dev link-local LLM gateway
(`http://169.254.169.254/gateway/llm/anthropic`) at startup before setting up
the Pi agent session.

- If the gateway is reachable (i.e. the process is running inside an exe.dev
  VM), it:
  - Sets a placeholder `"gateway"` runtime API key for the `anthropic` provider
    (the SDK requires a non-empty value; the gateway handles auth itself).
  - Registers an inline extension factory via `DefaultResourceLoader`'s
    `extensionFactories` option that calls
    `pi.registerProvider('anthropic', { baseUrl: GATEWAY_BASE_URL })`, pointing
    all Anthropic requests at the gateway endpoint.
- If the gateway is **not** reachable (local dev, CI, etc.), behaviour is
  unchanged: the `ANTHROPIC_API_KEY` environment variable is used as before.

The probe has a 2-second timeout so it fails fast outside of exe.dev (matching
the identical logic already used in `lib/llm-client.ts` for the chat API route).

## Why

The chat route (`app/api/chat/route.ts`) already uses the exe.dev gateway
transparently, making `ANTHROPIC_API_KEY` optional for chat on exe.dev. The Pi
evolve pipeline (pi-worker.ts) was the only remaining place that required a
real API key on exe.dev. This change brings the evolve pipeline in line with
the rest of the app — the gateway is used automatically in production with no
configuration required.
